havoc的请求格式以及请求流程

报文的请求格式比较简单，传输数据使用post方式，但是规定了POST数据的格式，首先需要在全部数据前加上一个12bytes的头，每4bytes代表不同的含义，分别为

[4 bytes] size - 整个报文的大小（包括12bytes的头）

[4 bytes] magic value - 魔术头（来分辨不同的第三方agent，对于同一个agent的请求数据需要使用同样的魔术头）

[4 bytes] agent ID - （随机生成的客户端id，区分不同的上线主机）

havoc对于一个请求的大概处理流程就是首先分析传入监听端口（http/s）的数据报文（POST方式），提取出其magic value，然后将一些信息以json格式转发到对应的agent handler脚本（通过websocket），然后生成相应的处理结果，然后再将处理结果返回到teamserver再由其转发到agent端，agent handler和teamserver建立websocket连接进行通信，agent端传输给teamserver的所有数据都会被转发到handler端进行处理，所以可以完全自定义数据流和处理流程

提供了官方的python api，可以实现agent端的注册，以及handler和teamserver的一些数据传输操作（websocket）

先用项目https://github.com/CodeXTF2/PyHmmm来简单理解api

demo：https://github.com/HavocFramework/Talon

agent handler处理脚本还是使用python api来写

用c编写可以加一些免杀手段，现在比较主流的

  • 反射DLL加载

    1.Brute Ratel C4作者的文章提出了原始反射dll项目存在的一些问题

    2.havoc作者给出了一个反射dll模板，支持nt，dos头擦除，以及分节的内存权限分配，解决了上面说到的问题

    3.还可以将正常的dll转化为一段shellcode，直接注入到进程中即可，使用SRDI或Donut，使用的也是反射dll加载技术，通过shellcode来反射加载而不是之前使用的通过导出函数的方式

    4.上述两个项目可能有特征被检查，也可以自己写，havoc的源码中提供了一个比较简单的shellcode加载器，有源码可以改改，在Teamserver/data/implants/Shellcode文件夹下，原理基本一样，最后将生成的shellcode附加到dll前面即可

  • indirect syscall

    在SysWhisper3作者文章中提到，直接的系统调用的栈内返回地址不在ntdll.dll中，可以使用比较简单的jmp方法跳转到ntdll.dll的某个随机syscall中，跳转的时候传入相应的系统调用号即可

Talon给的demo是一个简单的exe，这里将其编译为dll之后，在文件头加上反射加载的shellcode，以shellcode的形式注入进程当中，免杀效果比较好，目前只实现了简单的shell命令
